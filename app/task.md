задача:
получить данные по сети. сохранить их в БД. вернуть на UI.
в случае ошибки по сети - вернуть данные из БД.

реализация:
у нас будет 2 репозитория: Local и Remote. они не будут знать друг про друга.
каждый из них знает про свои Source'ы: Api и DB
каждый их них будет заниматься работой с сетью и с БД соответственно.
брать данные, мапить их и отдавать дальше. примитивными методами.

а координацией их работы будет заниматься domain-слой (UseCase). он будет знать про оба репозитория (точнее, зависеть от их интерфейсов).

репозитории не должны управлять логикой работы приложения. они тупые исполнители. коммуникаторы между Source'ами и Domain-слоем.
но их работа очень важна. благодаря им мы можем заменить Source, подправить код в Репозитории и все отсальное (domain и UI слои) не будут меняться.

обычно зависят от Source'ов, но могут и не зависеть от них. (если это не вызыает трудностей для тестов).
хоть это и не запрещается, но лучше не делать чтобы Репозитории зависели от других Репозиториев.

import kotlinx.coroutines.CoroutineScope

import kotlinx.coroutines.channels.Channel

import kotlinx.coroutines.flow.SharingStarted

import kotlinx.coroutines.flow.receiveAsFlow

import kotlinx.coroutines.flow.shareIn

import kotlinx.coroutines.launch



class SingleFlowEvent<T>(private val scope: CoroutineScope) {

    private val channel = Channel<T>(Channel.UNLIMITED)



    val flow = channel

        .receiveAsFlow()

        .shareIn(scope, SharingStarted.WhileSubscribed(), 0)



    fun emit(value: T) {

        scope.launch {

            channel.send(value)

        }

    }

}

в концепции MVI все знакомы с 2мя основными понятиями: State и Intent
State описывает состояние экрана. то, что должно на нем отображаться. храниться во ВьюМодели.
Intent - это действия пользователя, которые отправляются в ВьюМодель.

а что делать, если у нас есть какое-то однократно действие, которые нельзя описать как Стейт экрана:

    навигация на другой экрана
    показать Toast
    у нас есть большой список элементов. и надо программно проскролить к определенному элементу

это 3ий основной элемент событийно-ориентированного программирование - Event
для этого и нужен SingleFlowEvent.
во ВьюМодели в него будут отправлять типы событий, а подписчик уже будет их отрабатывать.

в концепции MVI мы должны создать свой класс событий:

sealed interface LoginEvent {

    data class ShowError(val error: String) : LoginEvent

    data object NavigateToSuccessScreen : LoginEvent

}

создать переменные во ВьюМодели

private val _event = SingleFlowEvent<LoginEvent>(viewModelScope)

val event = _event.flow

мы можем отправлять евенты во ВьюМодели следующим образом:

val result = // получение данных

if (result.isSuccess) {

    _event.emit(LoginEvent.NavigateToSuccessScreen)

} else {

    _event.emit(LoginEvent.ShowError(result.error.parseToString()))

}

на UI мы будем подписыаться на нужные нам события:

val event by remember { mutableStateOf(viewModel.event) }



LaunchedEffect(Unit) {

    event.filterIsInstance<LoginEvent.ShowError>().collect {

        // полаз Toast уведомления

    }

}



LaunchedEffect(Unit) {

    event.filterIsInstance<LoginEvent.NavigateToSuccessScreen>().collect {

        // навигация на другой экран

    }

}
